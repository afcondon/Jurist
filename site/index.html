Jurist demos

Small typed languages for mathematical things

Little languages for dynamical systems, symbolic expressions, optimization models, Petri nets, typed graphs — with the Julia scientific ecosystem as their production semantics. Each is a handful of PureScript types: cheap to mint, one per domain, and a description that typechecks is a well-formed object of that domain.

Like any well-made eDSL, each admits more than one interpreter — and they are not equals. The reference interpreter is plain PureScript, so it runs wherever PureScript runs: prototype on Node or the BEAM and check the description does what you mean. Julia runs that same reference interpreter too — byte-identically, below — but it also has interpreters no other runtime can offer: analytic Jacobians and stiff/DAE solvers, rigorously proven roots and optima, dynamics derived functorially from category theory. Portability is the development story; the Julia-only interpreters are the reason to deploy. And where two runtimes run the same interpreter, they agree to the last float:

The unglamorous part

Underneath is purejl, a from-scratch PureScript→Julia backend. It consumes CoreFn — the compiler's fully-elaborated core — so the whole language arrives by construction: typeclasses, row types, ADTs, the works (the row-typed SystemSpec machinery above is RowList-driven typeclass derivation, running on Julia). Core-library coverage: 50 built-in FFI shims (prelude, arrays incl. ST, strings, records, integers, numbers, …; Data.String.Regex is a deliberate loud-failure stub). Verified by differential testing against the JS backend as reference semantics: 422 of 426 outputs byte-identical, the other 4 documented representation choices (UTF-16 code units vs. codepoints).
