Portability is the development story; the Julia-only interpreters are the reason to deploy. And where two runtimes run the same interpreter, they agree to the last float — the Lorenz receipts at the top of the page.

Under the hood

In fact, Jurist isn't a minimal bridge to Julia — it's a full backend for the PureScript language, peer to the JavaScript, Erlang, and other backends, and built in the same way.

Whether you need all of PureScript or not, you won't be missing any of the language in your eDSL.

languageall of it, by construction — purejl consumes CoreFn, the compiler's fully-elaborated core, so typeclasses, row types, ADTs and the RowList machinery behind SystemSpec arrive pre-checked
pipelinestock purs → CoreFn → Julia source — the same shape as purerl and the JS backend
core libraries50 built-in FFI shims — prelude, effect, arrays (incl. ST), strings, records, integers, numbers, …
known gapData.String.Regex ships as a deliberate loud-failure stub
verificationdifferential testing against the JS backend as reference semantics: 422 of 426 outputs byte-identical
the other 4documented representation choices — UTF-16 code units vs. codepoints
your FFIan ffi-jl/ directory of Data_Module_foreign.jl files beside your project; user files win over the built-ins

A guiding force for the work is that if you have types in PureScript in, say, the browser, those types should mean the same thing on the Erlang or Go backend. If a core tenet of FP is to "make illegal states unrepresentable", then polyglot-PureScript aims to extend this across multiple runtimes — lessening the chances of a really gnarly cross-language bug.
