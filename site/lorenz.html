One description, three runtimes

Where possible the interpreter will run on other runtimes too — but for the Julia-specialty computations, you just write directly in the eDSL using Jurist. An example of something that can run on Node or the BEAM as well: the Lorenz butterfly, described once as a row-typed value (a misspelt field is a compile error), integrated by the same pure RK4 on all three runtimes — orbits agreeing byte for byte:

The PureScript you write

lorenz
  :: SystemSpec
       ( x :: NumExpr, y :: NumExpr, z :: NumExpr )
       ( sigma :: NumExpr, rho :: NumExpr, beta :: NumExpr )
lorenz = system \s p ->
  { x: p.sigma * (s.y - s.x)
  , y: s.x * (p.rho - s.z) - s.y
  , z: s.x * s.y - p.beta * s.z
  }

-- the pure denotation: plain PureScript, no FFI —
-- it compiles on ANY backend
orbit = integratePure lorenz
  { x: 1.0, y: 1.0, z: 1.0 }
  { sigma: 10.0, rho: 28.0, beta: 8.0 / 3.0 }
  0.01
  5000

The lambda's s.x is var "x" — it builds an AST, not a closure. One description, two denotations (ADR-0007): this pure RK4, and Julia's native compiled vector field.

What three runtimes compute

Julia byte-identical to the native RK4 over 5000 steps: true

The orbit every runtime agrees on (x–z projection)

Lorenz is chaotic — any difference in the arithmetic would diverge visibly within a few hundred steps. Bit-identity across three virtual machines is the strongest possible receipt that they ran the same description.
